## Versioning — do not manually bump

Finding #3's suggested fix ("bump on build") is **already automated**. When a PR
merges to `main`, the `Release on PR Merge` workflow
(`.github/workflows/merge-release.yml`) auto-bumps the workspace version (patch by
default), tags, publishes to npm, and attaches binaries to the GitHub release.

In this triage, the manual `0.4.67 → 0.4.68` bump stacked on top of the pipeline's
bump, so the release landed at `v0.4.69` instead of `v0.4.68`. No harm, but it
burned an extra patch number.

**Future fixes should not edit `package.json` versions (or `CHANGELOG.md`) by
hand** — let the merge-release pipeline do it. See `CONTRIBUTING.md` →
"Releases & Versioning".
